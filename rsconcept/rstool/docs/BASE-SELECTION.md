# Выбор основания

Краткий справочник для агентов: когда вводить `X#`, `C#` или использовать встроенную ступень `Z`. Термины — [DOMAIN.md](DOMAIN.md); поля конституент — [CONSTITUENTA.md](CONSTITUENTA.md).

## Выбор основания

| Что нужно выразить                                                      | Выбор | Почему                                                                       |
| :---------------------------------------------------------------------- | :---- | :--------------------------------------------------------------------------- |
| Предметные объекты без числового смысла: люди, документы, ситуации      | `X#`  | Элементы задаются интерпретацией и различаются только как элементы множества |
| Конечный именованный набор, где в формулах нужны порядок или арифметика | `C#`  | Элементы интерпретируются в КМ, но поддерживают сравнение и арифметику языка |
| Чистые целые: размеры, счётчики, индексы, `card`, рекурсия над числами  | `Z`   | Это встроенная ступень ЯРЭ; отдельная конституента не нужна                  |

```text
Нужно неопределяемое понятие?
  ├─ Сущности предметной области (люди, документы, ситуации) → X#
  └─ Числа / ранги / уровни
       ├─ Конечный именованный набор, интерпретируется в КМ → C#
       └─ Абстрактные целые, только вычисления (Z×Z, card, R{…}) → Z
```

Практическое правило: если число является свойством объекта, не кодируй объект через `C#`. Введи объект как `X#`, связь или атрибут как `S#`, а вычислимые выборки и проверки — как `D#` / `F#` / `P#`.

## Биективная переносимость

**Биективная переносимость** — независимость значения формального определения от биективной замены интерпретаций неопределяемых понятий: если согласованно переименовать все элементы `X1`, `C1`, `S1` (без слияний и потерь), результат вычисления зависит только от структуры, не от «имён». Семантический анализатор отсекает формулы, нарушающие переносимость; логическую непротиворечивость подтверждают отдельно — примером интерпретации в КМ.

Элементы **базисных множеств** принципиально несравнимы: нет встроенного порядка и арифметики. Допустимы только принадлежность (`ξ ∈ X1`), равенство внутри одной ступени (`a = b` при `a, b ∈ X1` — «тот же элемент», не порядок) и построения над множествами/кортежами. Запреты анализатора:

- `ξ < η` для `ξ, η ∈ X1` → `orderingNotSupported`;
- `X1 + 1`, `debool(X1) * 2` → `arithmeticNotSupported`;
- `debool(X1) = 1` → несовместимые ступени (`X1` и `Z`).

Необходимое условие переносимости — согласованная **типизация** всех идентификаторов ([TYPIFICATION.md](TYPIFICATION.md)): нарушение (`ξ ∈ X1` при `ξ : Z`, `S1 = S2` при разных ступенях) — ошибка ещё до вычисления.

Следствия для агента:

1. Предметные объекты без числового смысла — **`X#`**; числовые ограничения на них выражай через **`S#`**, **`F#`**, **`P#`**, а не через `<` на элементах `X#`.
2. Если формула «узнаёт» конкретные элементы базисного множества по способу, не заданному в схеме (скрытый порядок, кодирование атрибута константами) — вероятное нарушение переносимости или лишняя конституента.
3. Перед сохранением сомнительной формулы — `analyzeExpression`.

**Иллюстрация (родство):** при `X1 = {alice, bob, carol}` биективная замена на `{p1, p2, p3}` не меняет, кто чей родитель в `S1: ℬ(X1×X1)`, если пары переименованы согласованно. Формула `∀a,b∈X1 a < b` некорректна — порядок на людях в ЯРЭ не задан.

## Детали по вариантам

**`X#` — базисное множество.** Формальное определение пустое; конвенция фиксирует, что считается элементом. В КМ значения задаются явным перечислением. Элементы можно проверять на принадлежность и равенство внутри одной ступени, но нельзя упорядочивать или использовать в арифметике.

**`C#` — константное множество.** Формальное определение тоже пустое; значения задаются в КМ. В формулах допустимы сравнения и арифметика языка, включая согласованное смешивание с целочисленными литералами (`C1 ∪ {1}`, `ξ + 1`, `α < β`). Используй для шкал оценок, уровней допуска, номеров фаз процесса — когда эти значения являются частью модели.

**`Z` — встроенная ступень целых.** В схеме нет записи с `alias` для целых; `Z` используется прямо в типизациях и формулах (`Z×Z`, `[α∈Z]`, `ℬ(Z)`). В КМ `Z` отдельно не заполняется.

### `Z` и `C#`

| Вопрос                                 | `C#`                                          | `Z`                                                    |
| :------------------------------------- | :-------------------------------------------- | :----------------------------------------------------- |
| Есть ли конституента в списке схемы?   | да                                            | нет                                                    |
| Нужна ли интерпретация элементов в КМ? | да (`setModelValues`)                         | нет                                                    |
| Домен                                  | конечный именованный набор предметной области | бесконечные целые                                      |
| Типичная мотивация                     | «оценки 1–5», «уровни 0–3» как объекты модели | размеры, координаты, остатки, `card`, чистые алгоритмы |

Учебный пример **без** `X#` и `C#`: `examples/chocolate-nim/` — ширина и длина шоколадки как `Z×Z`, арифметика и `R{…}` только над `Z`.

## Минимальные примеры

**Базисное + отношение** (родство):

```text
X1   — люди (конвенция)
S1: ℬ(X1×X1)   — родитель–ребёнок
D1 := Pr1(S1)  — родители
```

**Константное** (шкала; элементы интерпретируются в модели):

```text
C1   — оценки 1…5 (конвенция)
P1 ::= [ξ∈C1] ξ ≥ 3
```

**Целые без конституенты** (размеры):

```text
S1: Z×Z          — пара (ширина, длина)
D1 := pr1(S1)    — ширина
F1 ::= [α∈Z] α - 1
```
