# RSLang и rstool

Краткий входной гайд для агентов. Детали — в связанных документах.

**RSLang** — формальная нотация концептуальных схем: понятия, отношения, операции, утверждения. Основа: принадлежность `x∈y`, теоретико-множественные и логические выражения, типизация.

**rstool** — API агента для сессий, редактирования схемы, анализа, диагностик, значений модели, вычисления, экспорта/импорта.

- Библиотека: `@rsconcept/rstool`.
- Анализатор: `@rsconcept/domain`.
- Stdio-обёртка: `npx rstool-wrapper`, JSON построчно.
- Node-клиент: `RSToolWrapperClient`.

## Быстрый контракт для агента

- Library API: `tool.applySchemaPatch(input, sessionId?)` — единственный путь правки схемы.
- Stdio/MCP: **плоские** `params` — поля метода и optional `sessionId` на одном уровне (без `params.input`).
- Состояние: `getSessionState()` (summary) или `getSessionState('full')`.
- Импорт/экспорт Portal: `importData`, `exportPortal({ kind })`.
- Модель: `setModelValues`, `evaluate`, `recalculateModel`.
- Перед сохранением сомнительной формулы — `analyzeExpression`.

## Воркфлоу

Все пошаговые сценарии собраны здесь. Справочные документы дают правила и детали.

### Редактирование и проверка схемы

Когда схема уже есть и нужны точечные правки, новые конституенты или проверка формул.

1. Продолжи текущую сессию или вызови `ensureSession`; большинство методов создают сессию лениво.
2. Развивай схему через `applySchemaPatch`: можно опускать `id` и `cstType` для стандартных alias `X/C/S/D/A/F/P`.
3. Для пачки конституент передавай все элементы вместе — `applySchemaPatch` упорядочит поставщиков раньше потребителей.
4. Перед patch в `applySchemaPatch` при сомнении вызывай `analyzeExpression`.
5. Исправляй диагностики по диапазону `from` / `to` в `definitionFormal` ([DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md)).
6. При сомнении в семантике собери маленькую КМ и проверь тестовые данные ([цикл КМ](#проверка-на-маленькой-км)).
7. Задай значения модели: `setModelValues`; вычисли: `evaluate`, `recalculateModel`.
8. `commitStep` или `commitMessage` в `applySchemaPatch`, когда состояние согласовано.
9. Сохрани: `exportSession` / `importData`.
10. Для загрузки в Portal: `exportPortal({ kind: 'schema' })` или `{ kind: 'model' }`.

### Оценка или правка КС из Portal

Когда пользователь даёт UI-ссылку вида `https://portal.acconcept.ru/rsforms/:id`,
не открывай SPA и не парси HTML. Преобразуй ссылку в REST API по правилам
[PORTAL-API.md](../../docs/PORTAL-API.md): метаданные `GET /api/rsforms/:id`,
полная КС `GET /api/rsforms/:id/details`, версия `GET /api/library/:id/versions/:version`.
Если id неизвестен — контекстный поиск `GET /api/library/context-search?q=...`, затем
метаданные по найденным id: `GET /api/library/by-ids?ids=...` (curl и поля поиска — в том же файле).

Portal JSON из `/details` не является форматом `exportSession`. Используй `importData(payload)` —
auto определит `portal-details` или `portal-schema`. Явный kind: `'portal-details'`, `'portal-schema'`, `'session'`.

### Концептуализация (КС из содержания источника)

Когда нужно построить или существенно развить КС по тексту ПО, требованиям или предметному описанию. Правила редукции, слоёв и паттернов — [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md).

1. Собери словарь: неопределяемые понятия, ключевые отношения, сценарий синтеза.
2. Добавь неопределяемые понятия `X#`, `C#`, `S#` с конвенциями из источника (не копируй дословно).
3. Проверь независимость неопределяемых понятий — они не выводятся друг из друга.
   Если отношение или множество выражается через уже введенные `X#`, `C#`, `S#`, оформляй его как `D#` / `F#` / `P#`.
4. Введи `F#` и `D#` для раскрытия проекций и функциональных связей в неопределяемых понятиях.
5. Добавь `A#` на структуры, выражающие ограничения предметной области. Возможно стоит ввести промежуточные производные термы и терм-функции.
6. Подбери промежуточные производные понятия `D#`, `F#`, `P#` так, чтобы выразить целевые понятия без использования не введенных в схеме понятий.
7. Перед каждым patch — `analyzeExpression`; при ошибках — [цикл диагностик](#цикл-диагностик).
8. Сомнительную семантику проверь на маленькой КМ ([цикл КМ](#проверка-на-маленькой-км), [MODEL-TESTING.md](../../docs/MODEL-TESTING.md)).
9. Перед показом результата пройди [чеклист ревью КС](#ревью-концептуальной-схемы).

Дальше — обычное [редактирование и проверка](#редактирование-и-проверка-схемы): commit, export.

### Цикл диагностик

Когда `analysis.success === false` или `listDiagnostics` не пуст. Коды и типичные ошибки — [DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md).

1. Запусти `analyzeExpression` (черновик без сохранения).
2. Прочитай `code`, `from`, `to`, `params` в `analysis.diagnostics` или через `listDiagnostics`.
3. Сопоставь `code` с исправлением в [DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md).
4. Исправь именно диапазон в `definitionFormal`.
5. Не повторяй вызов без изменения ввода.
6. После успеха — ещё один `applySchemaPatch` с исправленным элементом.

### Проверка на маленькой КМ

Когда синтаксис верен, но смысл формулы неочевиден. Подробности и форма данных — [MODEL-TESTING.md](../../docs/MODEL-TESTING.md).

1. Отдельная `createSession` или изолированная копия текущей сессии.
2. Только нужные поставщики: `X#`, `C#`, `S#` и проверяемые `D#` / `F#` / `P#` / `A#`.
3. `analyzeExpression` и `applySchemaPatch`.
4. Значения: `setModelValues`.
5. `evaluate`; сравни с ожидаемым `value`.
6. При нескольких зависимых определениях — `recalculateModel`.

Для регрессий вынеси проверку в скрипт или colocated `*.test.ts` (см. [MODEL-TESTING.md](../../docs/MODEL-TESTING.md)).

### Порядок объявления конституент

1. В начале схемы — `basic`, `constant`.
2. Родовые структуры производные и ключевые понятия, аксиомы и утверждения в произвольном порядке.
3. Конституенты в топологическом порядке.
4. Предпочитай размещать связанные с родовыми структурами конституенты вместе с этой структурой.

Правила валидации по типам — [CONSTITUENTA.md](../../docs/CONSTITUENTA.md).

## Формат ответа агента

В конце работы сообщи:

- что изменено в схеме или модели;
- есть ли активные диагностики;
- какой экспорт получен: сессия / схема для Portal / модель для Portal;
- какие проверки выполнены: анализ выражений / вычисление / пересчёт модели / тесты.

Не вставляй большой JSON в чат без запроса пользователя.

### Язык ответа

С пользователем говори на языке предметной области, а не контракта API. Английские идентификаторы (`definitionFormal`, `applySchemaPatch`, `cstType`...) уместны только в коде, patch-ах, JSON и при обсуждении самого API. В обычном тексте заменяй их терминами (полный словарь — [DOMAIN.md](../../docs/DOMAIN.md)):

| В API                             | В тексте ответа                     |
| :-------------------------------- | :---------------------------------- |
| `alias`                           | имя конституенты                    |
| `cstType`                         | тип конституенты                    |
| `definitionFormal`                | формальное определение (формула)    |
| `term`                            | термин                              |
| `definitionText`                  | текстовое определение               |
| `convention`                      | конвенция                           |
| `comment`                         | комментарий                         |
| `applySchemaPatch`                | правка схемы                        |
| `analyzeExpression`               | проверка / анализ выражения         |
| `listDiagnostics`, `diagnostics`  | диагностики                         |
| `setModelValues`, `getModelState` | интерпретация, значения модели      |
| `evaluate`, `recalculateModel`    | вычисление, пересчёт модели         |
| `commitStep`, `commitMessage`     | фиксация шага (ревизия)             |
| `exportSession` / `exportPortal`  | экспорт сессии / экспорт для Portal |
| `importData`                      | импорт данных                       |
| session, `sessionId`              | сессия                              |

Имена конституент (`X1`, `S2`, `D3`) и формулы RSLang — часть схемы, их приводи как есть; лучше вместе с термином: «терм `D3` (родитель)». Названия методов упоминай, только если пользователь спрашивает про rstool как инструмент.

Словарь выше — про поля rstool. Для **понятий схемы** каноничный источник — [DOMAIN.md](../../docs/DOMAIN.md).

## Чеклисты

### Сессия и rstool

- [ ] Используется текущая сессия (`getSessionState`) или явный `sessionId`, если сессий несколько.
- [ ] Для стандартных alias `X/C/S/D/A/F/P` можно не указывать `cstType` в `applySchemaPatch`.
- [ ] Для пачки конституент предпочитай один `applySchemaPatch` с массивом `items`.

### Перед показом, commit и export

- [ ] Схема проверена; ошибки исправлены.
- [ ] Диагностики обработаны до фиксации шага и экспорта.
- [ ] В тексте ответа нет английских имён полей и методов API — использованы термины из раздела [Язык ответа](#язык-ответа).
- [ ] Нет неофициальных ярлыков («базис», «базисный слой») — термины КС из [DOMAIN.md](../../docs/DOMAIN.md).

### Ревью концептуальной схемы

Перед завершением концептуализации или крупного изменения. Правила — [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md).

- [ ] Атрибутивная и родоструктурная экспликации не смешаны без явного запроса.
- [ ] Нет избыточной проверки принадлежности к собственной типизации (`x∈X1`, если `x` уже в `X1`).
- [ ] Неопределяемые понятия независимы; производные отношения вынесены в `D#` / `F#` / `P#`, а не объявлены новыми структурами.
- [ ] Покрыты главные сущности, роли, отношения, утверждения.
- [ ] В текстовых определениях используются только понятия из схемы.
- [ ] Повторяющиеся части формальных определений вынесены в производные понятия `D#` и `F#`.
- [ ] `R#` в `[…]` у `F#` / `P#` не считается пропущенной конституентой; радикалы не добавляют в схему как `X#` / `S#` / `D#`.
- [ ] Сомнительная семантика проверена на маленькой КМ при наличии тестовых данных.
