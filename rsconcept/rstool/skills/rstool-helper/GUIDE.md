# ЯРЭ и rstool

Краткий входной гайд для агентов. Детали — в связанных документах.

**ЯРЭ** (язык родоструктурной экспликации) — формальная нотация концептуальных схем: понятия, отношения, операции, утверждения. Основа: принадлежность `x∈y`, теоретико-множественные и логические выражения, типизация.

**rstool** — API агента для сессий, редактирования схемы, анализа, диагностик, значений модели, вычисления, экспорта/импорта.

- Библиотека: `@rsconcept/rstool`.
- Анализатор: `@rsconcept/domain`.
- Stdio-обёртка: `npx rstool-wrapper`, JSON построчно.
- Node-клиент: `RSToolWrapperClient`.
- MCP: `@rsconcept/rstool-mcp` (tools в `snake_case` — [REFERENCE.md](REFERENCE.md)).

## Быстрый контракт для агента

- Library API: `tool.applySchemaPatch(input, sessionId?)` — основной путь правки схемы; `synthesize` — встраивание конституент из другой сессии (синтез с таблицей отождествлений).
- Stdio/MCP: **плоские** `params` — поля метода и optional `sessionId` на одном уровне (без `params.input`).
- Состояние: `getSessionState()` (summary) или `getSessionState('full')`.
- Импорт/экспорт Portal: `importData`, `exportPortal({ kind })`.
- Модель: `setModelValues`, `evaluate`, `recalculateModel`.
- Перед сохранением сомнительной формулы — `analyzeExpression`.

## Границы правок: не трогай пакет rstool без запроса

Задача по КС, Portal или источнику пользователя **не означает** правку дерева `rsconcept/rstool/`.
Пакет — библиотека и справочник; каталог `examples/` — **штатные** демо пакета, а не место для разовых скриптов агента.

Правки в rstool уместны только если пользователь **явно** просит, например:

- доработать API, stdio, контракт, тесты rstool;
- добавить **общий** пример в пакет;
- обновить skill/docs rstool (в т.ч. это руководство).

### Как запускать rstool, не кладя скрипт в пакет

Из корня workspace Portal (после `pnpm install` и сборки `@rsconcept/domain`):

```bash
pnpm --filter @rsconcept/rstool exec tsx path/to/your/build-schema.ts
```

В скрипте импортируй API пакета (`@rsconcept/rstool` или относительный путь к `rsconcept/rstool/src` только если скрипт лежит вне пакета и это осознанный выбор). Либо используй stdio/MCP без отдельного файла.

## Воркфлоу

Все пошаговые сценарии собраны здесь. Справочные документы дают правила и детали.

### Редактирование и проверка схемы

Когда схема уже есть и нужны точечные правки, новые конституенты или проверка формул.

1. Продолжи текущую сессию или вызови `ensureSession`; большинство методов создают сессию лениво.
2. Развивай схему через `applySchemaPatch`: можно опускать `id` и `cstType` — они выводятся из префикса `alias` (`X/C/S/D/F/P/A/T/N`).
3. Для пачки конституент передавай все элементы вместе — `applySchemaPatch` упорядочит поставщиков раньше потребителей.
4. Перед patch в `applySchemaPatch` при сомнении вызывай `analyzeExpression`.
5. Исправляй диагностики по диапазону `from` / `to` в `definitionFormal` ([DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md)).
6. При сомнении в семантике собери маленькую КМ и проверь тестовые данные ([цикл КМ](#проверка-на-маленькой-км)).
7. Задай значения модели: `setModelValues`; вычисли: `evaluate`, `recalculateModel`.
8. `commitStep` или `commitMessage` в `applySchemaPatch`, когда состояние согласовано.
9. Сохрани: `exportSession` / `importData`.
10. Для загрузки в Portal: `exportPortal({ kind: 'schema' })` или `{ kind: 'model' }`.

### Оценка или правка КС из Portal

Когда пользователь даёт UI-ссылку вида `https://portal.acconcept.ru/rsforms/:id`,
не открывай SPA и не парси HTML: преобразуй ссылку в REST-запрос по правилам
[PORTAL-API.md](../../docs/PORTAL-API.md) — там пути, контекстный поиск, правила доступа и curl.

Portal JSON из `/details` не является форматом `exportSession`. Используй `importData(payload)` —
kind `auto` (default) определит `portal-details` или `portal-schema`. Явный kind: `'portal-details'`, `'portal-schema'`, `'session'`.

Если задача — **оценка** схемы из Portal, а не правка: после импорта иди в [ревью / оценка](#ревью--оценка-готовой-кс), не начинай массовый refactor.

### Концептуализация (КС из содержания источника)

Когда нужно построить или существенно развить КС по тексту ПО, требованиям или предметному описанию. Правила редукции, слоёв и паттернов — [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md).

1. Собери словарь: неопределяемые понятия, ключевые отношения, сценарий синтеза.
2. Добавь независимые неопределяемые понятия `X#`, `C#`, `S#` с конвенциями из источника (не копируй дословно). Для выбора `X#` vs `C#` vs `Z` — [BASE-SELECTION.md](../../docs/BASE-SELECTION.md).
3. Всё, что выводится из уже введённых понятий, оформляй как `D#` / `F#` / `P#`, а не как новое `X#` / `C#` / `S#`.
4. Вводи промежуточные `D#`, `F#`, `P#` для проекций, функциональных связей и повторяющихся фрагментов формул. Если зависимость имеет вид «для `ξ` вычислить значение», оформляй её как `F#`, а не как множество пар `I{(ξ,σ)|…}` ([CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md), «Терм-функция вместо множества пар»).
5. Добавь `A#` и `T#` после терминов, на которые они опираются.
6. Перед каждым patch — `analyzeExpression`; при ошибках — [цикл диагностик](#цикл-диагностик).
7. Сомнительную семантику проверь на маленькой КМ ([цикл КМ](#проверка-на-маленькой-км), [MODEL-TESTING.md](../../docs/MODEL-TESTING.md)).
8. Перед показом результата пройди [чеклист ревью КС](#ревью-концептуальной-схемы).

Дальше — обычное [редактирование и проверка](#редактирование-и-проверка-схемы): commit, export.

### Ревью / оценка готовой КС

Когда пользователь просит **оценить**, **проревьюить**, **найти проблемы** в уже существующей схеме (сессия, JSON, ссылка Portal) — и **не** просит сразу переписать её.

**Правило:** без явного запроса на правку не делай массовый рефакторинг и не заменяй схему «улучшенной версией». Сначала отчёт; правки — только по согласованию или если пользователь сразу сказал «исправь».

1. Загрузи схему: текущая сессия, `importData`, или Portal → [PORTAL-API.md](../../docs/PORTAL-API.md).
2. `listDiagnostics()` — раздели `expression` / `schema` / `model`; зафиксируй ошибки и предупреждения.
3. `getSessionState('summary')` — обзор состава; при необходимости `full` для формул.
4. Структурный проход по [чеклисту ревью](#ревью-концептуальной-схемы) и [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md):
   - неопределяемые vs производные (`S#` там, где должно быть `D#`/`F#`/`P#`);
   - `X#` / `C#` / `Z` ([BASE-SELECTION.md](../../docs/BASE-SELECTION.md));
   - терм-функции vs множества пар;
   - радикалы `R#` не объявлены конституентами;
   - покрытие задачи vs избыточность.
5. Severity в отчёте:
   - **блокер** — диагностики `error`, схема не анализируется / ломает вычисление;
   - **важно** — нарушение правил проектирования (лишнее `S#`, график вместо `F#`, смешение экспликаций);
   - **замечание** — стиль, именование, возможные упрощения, пробелы покрытия.
6. Ответ — по [шаблону ревью](#шаблон-ответа-ревью). Предлагай конкретные правки списком; применяй их только по запросу.

### Цикл диагностик

Когда `analysis.success === false` или `listDiagnostics` не пуст. Коды и типичные ошибки — [DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md); короткие «было → стало» — [EXAMPLES.md](EXAMPLES.md).

1. Вызови `listDiagnostics()` — смотри `kind`: `expression` (формула), `schema` (омонимы, дубликаты, термин/конвенция), `model` (интерпретация).
2. Для черновика формулы — `analyzeExpression` (без сохранения).
3. Прочитай `name`, `severity`, `alias`, `from`, `to`, `params` в `analysis.diagnostics` или через `listDiagnostics({ kind })`.
4. Сопоставь `name` (или `code`) с исправлением в [DIAGNOSTICS.md](../../docs/DIAGNOSTICS.md).
5. Исправь именно диапазон в `definitionFormal` (expression) или метаданные/данные модели (schema/model).
6. Не повторяй вызов без изменения ввода.
7. После успеха — ещё один `applySchemaPatch` / `setModelValues` с исправлением.

### Проверка на маленькой КМ

Когда синтаксис верен, но смысл формулы неочевиден. Подробности и форма данных — [MODEL-TESTING.md](../../docs/MODEL-TESTING.md).

1. Отдельная `createSession` или изолированная копия текущей сессии.
2. Только нужные поставщики: `X#`, `C#`, `S#` и проверяемые `D#` / `F#` / `P#` / `A#`.
3. `analyzeExpression` и `applySchemaPatch`.
4. Значения: `setModelValues`.
5. `evaluate`; сравни с ожидаемым `value`.
6. При нескольких зависимых определениях — `recalculateModel`.

Для регрессий вынеси проверку в скрипт или colocated `*.test.ts` (см. [MODEL-TESTING.md](../../docs/MODEL-TESTING.md)).

### Порядок объявления конституент

В начале схемы — `basic` и `constant`, затем независимые `structure`, затем производные (`term`, `function`, `predicate`, `axiom`, `statement`) в топологическом порядке. Производные при родовой структуре держи рядом с ней, если это не ломает зависимости.

Правила валидации по типам — [CONSTITUENTA.md](../../docs/CONSTITUENTA.md); разворачивание понятий при проектировании — [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md).

## Формат ответа агента

Выбери шаблон по типу задачи. Не вставляй большой JSON в чат без запроса пользователя.

### Сборка / правка схемы

В конце работы сообщи:

- что изменено в схеме или модели;
- есть ли активные диагностики;
- какой экспорт получен: сессия / схема для Portal / модель для Portal;
- какие проверки выполнены: анализ выражений / вычисление / пересчёт модели / тесты.

### Шаблон ответа: ревью

1. **Краткий вердикт** (1–2 предложения): пригодна ли схема к задаче, главный риск.
2. **Диагностики:** число ошибок / предупреждений по `expression` / `schema` / `model` (без полного JSON).
3. **Находки** списком: блокер → важно → замечание; у каждой — имя конституенты и суть.
4. **Покрытие:** что из задачи отражено; чего не хватает.
5. **Рекомендации:** конкретные правки; не применяй их без запроса.
6. **Проверки:** что уже сделано (import, listDiagnostics, выборочный анализ).

### Шаблон ответа: диагностики

- какие конституенты / формулы затронуты;
- `name` диагностики и смысл простым языком;
- что изменено в формальном определении (или почему правка не нужна);
- повторная проверка: чисто / осталось.

### Шаблон ответа: объяснение формулы

- роль конституенты и термин;
- формула как есть;
- по шагам: что вычисляется / утверждается;
- на какие понятия опирается; типизация.

### Язык ответа

С пользователем говори на языке предметной области, а не контракта API. Английские идентификаторы (`definitionFormal`, `applySchemaPatch`, `cstType`...) уместны только в коде, patch-ах, JSON и при обсуждении самого API. В обычном тексте заменяй их терминами (полный словарь — [DOMAIN.md](../../docs/DOMAIN.md)):

| В API                             | В тексте ответа                     |
| :-------------------------------- | :---------------------------------- |
| `alias`                           | имя конституенты                    |
| `cstType`                         | тип конституенты                    |
| `definitionFormal`                | формальное определение (формула)    |
| `term`                            | термин                              |
| `definitionText`                  | текстовое определение               |
| `convention`                      | конвенция                           |
| `comment`                         | комментарий                         |
| `applySchemaPatch`                | правка схемы                        |
| `synthesize`                      | встраивание / синтез схем           |
| `analyzeExpression`               | проверка / анализ выражения         |
| `listDiagnostics`, `diagnostics`  | диагностики                         |
| `setModelValues`, `getModelState` | интерпретация, значения модели      |
| `evaluate`, `recalculateModel`    | вычисление, пересчёт модели         |
| `commitStep`, `commitMessage`     | фиксация шага (ревизия)             |
| `exportSession` / `exportPortal`  | экспорт сессии / экспорт для Portal |
| `importData`                      | импорт данных                       |
| session, `sessionId`              | сессия                              |

Имена конституент (`X1`, `S2`, `D3`) и формулы ЯРЭ — часть схемы, их приводи как есть; лучше вместе с термином: «терм `D3` (родитель)». Названия методов упоминай, только если пользователь спрашивает про rstool как инструмент.

Словарь выше — про поля rstool. Для **понятий схемы** каноничный источник — [DOMAIN.md](../../docs/DOMAIN.md).

## Чеклисты

### Сессия и rstool

- [ ] Используется текущая сессия (`getSessionState`) или явный `sessionId`, если сессий несколько.
- [ ] Для стандартных `X/C/S/D/F/P/A/T/N` можно не указывать `cstType` в `applySchemaPatch`.
- [ ] Для пачки конституент предпочитай один `applySchemaPatch` с массивом `items`.

### Перед показом, commit и export

- [ ] Схема проверена; ошибки исправлены.
- [ ] Диагностики обработаны до фиксации шага и экспорта.
- [ ] В тексте ответа нет английских имён полей и методов API — использованы термины из раздела [Язык ответа](#язык-ответа).
- [ ] Нет неофициальных ярлыков («базис», «базисный слой») — термины КС из [DOMAIN.md](../../docs/DOMAIN.md).

### Ревью концептуальной схемы

Перед завершением концептуализации, крупного изменения или как проход в [воркфлоу ревью](#ревью--оценка-готовой-кс). Правила — [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md).

- [ ] Атрибутивная и родоструктурная экспликации не смешаны без явного запроса.
- [ ] Нет избыточной проверки принадлежности к собственной типизации (`x∈X1`, если `x` уже в `X1`).
- [ ] Неопределяемые понятия независимы; производные отношения вынесены в `D#` / `F#` / `P#`, а не объявлены новыми структурами.
- [ ] `X#` — только предметные сущности; `C#` — только если нужны порядок/арифметика над именованным конечным набором в КМ; чистая целочисленная математика — `Z` без лишней конституенты ([BASE-SELECTION.md](../../docs/BASE-SELECTION.md)).
- [ ] Покрыты главные сущности, роли, отношения, утверждения.
- [ ] В текстовых определениях используются только понятия из схемы.
- [ ] Повторяющиеся части формальных определений вынесены в производные понятия `D#` и `F#`.
- [ ] Вычисляемые зависимости «аргумент → значение» оформлены как `F#` с вызовами, а не как пары / тройки + `Pr`/`Fi` (см. [CONCEPTUAL-SCHEMA.md](../../docs/CONCEPTUAL-SCHEMA.md), «Терм-функция вместо множества пар»).
- [ ] Пары / `I{(ξ,σ)|…}` оставлены только для истинных отношений, графиков как объектов или многозначных связей.
- [ ] `R#` в `[…]` у `F#` / `P#` не считается пропущенной конституентой; радикалы не добавляют в схему как `X#` / `S#` / `D#`.
- [ ] Сомнительная семантика проверена на маленькой КМ при наличии тестовых данных.
- [ ] При чистом ревью: отчёт по [шаблону](#шаблон-ответа-ревью) выдан **до** любых массовых правок.
