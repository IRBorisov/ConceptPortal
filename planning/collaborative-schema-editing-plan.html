Collaborative Schema Editing Plan

Plan for multiple users editing the same Portal schema with synced changes, visible presence, soft locks, conflict detection, and chat scoped to a schema.

Target: RSForm schema editor first Editing model: presence + soft locks Chat access: anyone who can read Transport: websocket channel per schema

Current Baseline

What exists

  • RSForm data is loaded through REST endpoints such as /api/rsforms/:id/details.
  • Edits are granular REST mutations, and successful responses replace TanStack Query cache through updateRSForm.
  • Access control already distinguishes owners, editors, readers, and admins via LibraryItem and Editor.
  • Explicit versions exist as manual snapshots, not as an operation history.

What is missing

  • No websocket, Server-Sent Events, or collaborative transport exists today.
  • No optimistic concurrency: concurrent REST writes are effectively last-write-wins.
  • No presence, active editor list, field locks, or edit-session heartbeat.
  • No schema-scoped chat model or message APIs.
First release should avoid CRDT/OT character-level merging. The existing editor saves meaningful schema operations, so collaboration should start at operation and constituent scope with conflict prevention and clear recovery.

Product Shape

Presence

Show who is viewing the schema, who is editing, and which constituent or tab they are focused on.

Soft Locks

When an editor starts editing a constituent or metadata form, other users see that area as occupied and can choose to wait or override.

Synced Changes

After a mutation commits, other connected clients receive the updated revision and refresh or patch their local schema cache.

Conflict Detection

Each write carries a base revision. Stale writes are rejected or require an explicit reload/override flow.

Schema Chat

Readers and editors can send messages in a schema room, optionally linked to a constituent or current revision.

History

Chat history is persisted. Schema operation history can be added later, separate from existing manual versions.

Backend Plan

Area Change Notes
Realtime transport Add ASGI and Django Channels, with websocket rooms keyed by schema:{library_item_id}. Production will need nginx websocket proxying and a channel layer. Redis is the usual choice.
Revisioning Add a monotonic revision integer to LibraryItem or a schema-specific state table. All mutating RSForm endpoints should require or accept base_revision.
Conflict checks Inside existing transaction.atomic() mutations, compare request base revision with current revision. If stale, return 409 Conflict with current revision and a summary of changed areas.
Presence Create ephemeral sessions with heartbeat: user, schema item, tab, selected constituent, editing scope, last seen. May live in channel memory/Redis first; persist only if needed for audit.
Soft locks Create lock records for scopes like schema:metadata, cst:{id}, attribution:{id}. Locks expire automatically. Override should be logged and broadcast.
Chat Add SchemaChatMessage: schema item, author, body, created time, optional constituent, optional revision. Read permission follows schema read access. Writing chat also follows read access.
Permissions Reuse existing can_read_library_item for chat and presence visibility, and ItemEditor for edit locks. Also enforce read_only server-side before accepting schema mutations.

Frontend Plan

Collaboration Provider

Add a provider mounted inside RSForm pages, connected to the active schema id and current user. It owns websocket lifecycle, reconnects, room membership, and event dispatch.

Initial integration points: schema-edit-state.tsx, schema-edit-context.tsx, and updateRSForm().

Cache Updates

Remote committed mutation events should update or invalidate the schema query. If the user has unsaved local changes, show a non-destructive conflict banner instead of replacing the draft.

Lock UX

Forms request soft locks when they become dirty. Other editors see badges and disabled or warning states on the same constituent or metadata section.

Chat Panel

Add a right-side panel or tab for schema chat. Messages can reference the active constituent and revision so discussion remains tied to schema context.

Event Contract

Event Direction Payload
presence.joined, presence.updated, presence.left Server to clients User id/name, schema id, selected tab, selected constituent, editing scope.
lock.request, lock.granted, lock.released, lock.overridden Both Lock scope, owner, expiry, reason, optional override metadata.
schema.mutation_committed Server to clients Schema id, mutation kind, changed scopes, new revision, optional full RSForm DTO.
schema.conflict Server to client Submitted base revision, current revision, changed scopes, suggested recovery action.
chat.message_created Both Message id, author, body, created time, schema id, optional constituent id and revision.

Phased Delivery

  1. Foundation: add server-side read_only enforcement, schema revisioning, and stale-write 409 Conflict responses for RSForm mutations.
  2. Realtime shell: introduce websocket room auth, connect/disconnect, heartbeat, and presence display without changing edit behavior.
  3. Synced commits: broadcast successful schema mutations and refresh other clients through the existing query cache path.
  4. Soft locks: add editing scope locks, expiry, override, and frontend badges/warnings on metadata and constituents.
  5. Schema chat: persist chat messages, load history over REST, send new messages over websocket, and attach messages to optional constituent context.
  6. Broaden coverage: extend the same collaboration shell to RSModel and OSS editing surfaces after RSForm behavior is stable.

Risks And Decisions

Main Risks

  • Propagation from RSForm edits into OSS may require broadcasting affected downstream items.
  • Full DTO refreshes are simple but may feel heavy if schema documents are large.
  • Soft locks prevent most conflicts, but override flows must be explicit and auditable.
  • Websocket auth must align with existing session-cookie and CSRF assumptions.

Recommended Defaults

  • Use full-schema refresh after remote commits for v1, then optimize to scoped patches later.
  • Make locks advisory but visible, with auto-expiry and owner/editor/admin override.
  • Allow chat for anyone who can read the schema, matching the selected product scope.
  • Keep chat history separate from schema versions, but store message revision for context.
The lowest-risk v1 is not simultaneous text merging. It is realtime awareness, server-side stale-write protection, fast propagation of committed edits, and a schema room where users can coordinate work.
