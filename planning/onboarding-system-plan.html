Portal Onboarding System

A reusable tour engine — registry, persisted state, and spotlight/popover UI — mounted in both the main app shell and the Sandbox shell. The Sandbox intro tour ships as its first consumer; any main-UI page can register its own tour later with zero engine changes.

Frontend Tour engine Main UI + Sandbox Onboarding Extensible registry

Goal

Build one onboarding system instead of a Sandbox-only feature. The engine handles spotlighting, step navigation, persistence, and re-offering after version bumps; individual tours are pure data plus localized content. V1 ships the engine and the Sandbox intro tour, which teaches that a schema defines concepts and relationships while a model provides concrete values and evaluation results.

Engine vs. Tours

src/features/onboarding — the engine

Owns tour/step types, the tour registry, the persisted onboarding store, the TourHost visual layer, and the anchor resolver. Knows nothing about any specific tour.

tours/ — pure data consumers

Each tour is a definition object: id, version, ordered steps with anchors, optional onEnter hooks for tab/route switching, and per-locale content modules (content.en.tsx / .ru.tsx / .fr.tsx) following the help manuals convention.

TourHost mounted in both shells

Added to layout-main.tsx and layout-sandbox.tsx beside AppShellDialogHosts. Renders nothing unless a tour is active or a first-run trigger fires on its declared route.

data-tour anchors in existing UI

Inert attributes on stable UI elements (Sandbox tab chrome, key panels). The anchor resolver waits for elements, scrolls them into view, and tracks their rects; missing anchors skip the step with a warning instead of blocking the tour.

Persistence Model

  • New zustand store persisted as portal.onboarding with version + migrate, following the library-search.ts pattern.
  • Per-tour record: status (pending / skipped / done), seenVersion, and resumeStep.
  • Browser-local only — same behavior for anonymous Sandbox users and authenticated users. No backend changes.
  • Bumping a tour's version in its definition re-offers it after UX changes; Escape persists the resume point instead of losing progress.

Key Decisions

  • Custom tour layer over existing Base UI Popover/PopoverAnchor primitives plus a body-portal spotlight overlay (box-shadow cutout); no third-party tour library.
  • New --z-index-tour above modal (60) in index.css so steps can spotlight over dialogs.
  • Steps without an anchor render as centered modal steps (welcome/finish); steps with anchors get spotlight + popover.
  • Short strings (buttons, progress) as tx.onboarding.*; long step prose in per-locale tour content files, per frontend i18n rules.
  • V1 is read-only over the starter bundle — no mutation steps, so the tour cannot trap users in an invalid state.
  • No changes to Sandbox Dexie persistence, schema/model APIs, or starter bundle format.

Acceptance Criteria

  • The engine has no Sandbox-specific code; adding a main-UI tour requires only a new definition in tours/ plus anchors.
  • First-time Sandbox visitors see a skippable welcome step; Start advances across Passport, List, Concept, Graph, Data, and Evaluation via existing tab navigation.
  • Skip and Done persist in portal.onboarding and prevent the first-run flow from reappearing; version bumps re-offer the tour.
  • Escape dismisses and stores a resume point; the tour is fully keyboard accessible.
  • Missing anchors skip their step gracefully instead of blocking.
  • Locale parity tests pass after adding onboarding strings.

Testing Plan

Engine state

Vitest: version bump re-offer, skip/done/resume transitions, store migrate, registry validation (unique step ids, declared anchors).

Flow

Playwright: first visit shows welcome, Start advances across tabs, Skip persists (assert portal.onboarding in localStorage).

Regression

Completed users see nothing; existing Sandbox editing and main UI behavior are unaffected beyond inert anchor attributes.

Scope Boundary

The onboarding system teaches through the current editors. It must not change Sandbox Dexie persistence, schema/model mutation contracts, the starter bundle format, or backend behavior. Only the engine module, shell mount points, inert anchor attributes, one CSS variable, and i18n strings are touched. V1 ships the Sandbox intro tour only; main-UI tours are follow-up consumers of the same engine.
