Constituenta naming & type structure

Overview of how Constituenta is used across @rsconcept/domain, and whether the pain is the domain noun itself or the capability slices that reuse it.

Scope: domain (+ frontend / rstool mirrors) Status: option 1 implemented Date: 2026-07-11 domain 3.0.0
Done (option 1). Kept Constituenta as the entity noun. Renamed capability slices to concern/task *Fields names, aligned with ConceptCore’s RSConcept / TextConcept split (slices named by concern, never *Concept adjectives).
Original verdict. Keep Constituenta as the canonical entity name (locked in CONTEXT.md). The overload comes from capability slices named like fake subtypes (*Constituenta). Prefer renaming those slices first; optionally split the fat entity later. Do not rename the domain noun.

What’s canonical

CONTEXT.md locks конституента → Constituenta as the English domain alias. That should stay for the real entity and for UI/API surfaces that talk about one constituenta.

Keep the noun

Entity, UI labels, API verbs (createConstituenta), and docs should continue to say Constituenta. This matches the domain glossary and Russian product language.

Where it hurts

Almost every partial shape also ends in *Constituenta, so readers treat “fields for alias allocation” as “a kind of constituenta.” That is the naming smell.

ConceptCore precedent

In GH-ConceptCore, schema items are split by concern (RSConcept, TextConcept, ParsingInfo, ConceptRecord) and capabilities live on facets (CstList, IdentityManager, Thesaurus). The entity noun (“constituent”) appears in comments / JSON wire format, not on slice type names. Portal’s TypeScript equivalent is duck-typed *Fields constraints for pure transforms.

Slices constrain transforms; they are not Pick<Constituenta, …>. The full entity requires many computed fields, so incomplete payloads need duck-typed interfaces.

The fat entity

Constituenta in rsform.ts mixes several concerns (~25+ fields):

Persistent

id, alias, cst_type, terms, definitions, convention, typification_manual, attributes, crucial flag.

Analyzed / computed

analysis, diagnostics, effectiveType, status, cst_class, template / simple-expression flags.

Inheritance / spawn

parent schema indices, inherited flags, spawner / spawn paths and aliases.

Capability slices

Pure transform helpers in rsform-api/ take structural duck-types instead of the full entity.

Current name Real job Fields Smell
OrderableConstituenta Formal dependency graph / restore order id, alias, cst_type, definition_formal noun misuse
ResolvableConstituenta Resolve term / definition entity references id, alias, term_*, definition_raw / resolved noun misuse
AliasTypedConstituenta Allocate / validate aliases by type alias, cst_type noun misuse
ConstituentaMappableFields In-place alias remapping alias + formal / text fields clearer pattern
SynthesizableConstituenta Inline synthesis / schema embedding intersection of the above + id compound noun

ConstituentaMappableFields already broke the *Constituenta pattern — and reads better because of it.

Where overuse shows up

Identifier pressure

Across domain / frontend / rstool there are roughly ~90 distinct identifiers containing Constituenta. The full entity name dominates (~400 hits). Slice names are few but noisy.

Parallel families

  • frontend: ConstituentaBasicsDTO, CreateConstituentaDTO, …
  • rstool: ConstituentaDraft / ConstituentaState (camelCase, different lifecycle)

Same noun, different contracts — fine if intentional, confusing when reading APIs.

Comments also flip between “constituent”, “constituenta”, and Cst*. Worth normalizing wording even if type names stay.

Directions

1. Keep the noun, rename slices recommended first

Keep Constituenta for the entity. Rename capability types to task / capability language, preferably *Fields or role nouns:

  • OrderableConstituenta → OrderableFields or FormalOrderItem
  • ResolvableConstituenta → TextResolvableFields
  • AliasTypedConstituenta → AliasTypedFields
  • SynthesizableConstituenta → SynthesizableFields (or keep as intersection alias)
  • Align with existing ConstituentaMappableFields

Low blast radius (mostly domain + a handful of generics). No CONTEXT.md change.

2. Split the fat entity optional later

Introduce layered types, e.g. ConstituentaCore (persistable) → ConstituentaAnalyzed → full Constituenta with inheritance / spawn. DTOs / Draft / State could map onto layers instead of inventing sibling nouns.

Higher cost; do only if the god-object itself is the real pain.

3. Rename Constituenta → Constituent / Concept / … not recommended

Fights the domain glossary and churns hundreds of call sites for little clarity.

Decision fork

Chose C then A: rename slices now; leave fat-entity layering for a later pass if needed.

A — Slice names done

Capability types feel like fake “kinds of constituenta.” Cheap win; matches direction 1.

B — Fat entity deferred

The full Constituenta interface mixes too many concerns. Needs direction 2.

C — Both selected

Rename slices first as a cheap win, then decide whether layered entity types are worth the follow-up.
