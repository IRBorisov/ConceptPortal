Help Tour UX Improvement Plan

Convert the current read-only walkthrough into reliable, accessible, task-oriented guidance while preserving its strong lazy-loading, localization, persistence, route-resume, and nested-tour architecture.

Product conclusion. The current engine behaves as a slideshow over an inert application, but its copy frequently asks users to click, select, drag, and use shortcuts. The target experience should combine explanatory modal steps with interactive practice steps that advance from real user actions.

Target experience

1. Invite

Offer help after page readiness with a clear purpose, step count, approximate duration, Start, and Not now. Avoid opening a blocking full-screen tour merely because a route was visited.

2. Practice

Keep highlighted controls operable on interactive steps and advance from successful actions. Retain modal/inert behavior only where interaction is unsafe or unnecessary.

3. Recover

Make Close for now, Skip tour, unavailable targets, route changes, and loading failures explicit. Preserve progress without surprising permanent choices.

Prioritized findings

P0

The tour teaches actions users cannot perform · effort L

Every active tour marks #root inert and catches pointer input over the page. Highlighted controls are visual only, even when the copy says click, drag, select, or press.

Plan: introduce explanatory and interactive step modes with action-based completion.

Evidence: tour-host.tsx:290–305, 389–405; multiple tour content files.

P0

Escape has contradictory meanings · effort M

The host globally captures Escape to dismiss the tour while list and graph steps teach Escape as clear selection.

Plan: add Close for now; reserve Skip tour for permanent opt-out; do not globally consume Escape during interactive steps.

Evidence: tour-host.tsx:271–287; constituents-list and term-graph content.

P0

Missing targets silently erase steps · effort M

Anchors get only 800 ms to appear. Delayed, conditional, read-only, or absent controls silently advance the tour and can cause an incomplete run to be marked done.

Plan: resolve visible targets using readiness signals and MutationObserver; display Retry / Continue when unavailable; never auto-complete a silently skipped run.

Evidence: tour-host.tsx:163–179; anchors.ts:3–63.

P0

Step transitions can spotlight the previous control · effort S

The old tracked rectangle remains active while the next anchor resolves.

Plan: clear resolved anchor and tracked rectangle immediately on tour/step changes; render a neutral loading position until the new target is ready.

Evidence: tour-host.tsx:56–60, 216–238.

P0

A failed tour chunk can lock the app · effort S

Inert state depends on activeTourID, while the card requires successfully loaded content. A lazy-load failure can leave no visible card and an unusable application.

Plan: clear failed tours, restore app and tooltip state, show a retryable message, and apply inert only when a modal is successfully rendered.

Evidence: tour-host.tsx:117–134, 290–322; tours/index.ts:125–139.

P1

First-visit auto-start is interruptive · effort M

Sandbox and Library immediately open blocking tours. Version bumps also re-offer tours to users who previously skipped them.

Plan: use a lightweight invitation and preserve deliberate opt-outs.

P1

Accessibility announcements and motion preferences are incomplete · effort S

Step title/body are not associated with the dialog or reliably announced on change. Transitions ignore reduced-motion preferences.

Plan: add aria-labelledby, aria-describedby, polite step announcements, safe focus restoration, primary-action focus, and reduced-motion styles.

P1

Nested tours hide the real commitment · effort S

The eight-step Sandbox tour can branch into three-to-six-step subtours. The generic Details action does not identify the destination or length, and progress resets without context.

Plan: use labels such as “Explore List · 5 steps”, display a breadcrumb, and provide Return to overview.

P1

Contextual help entry is fragile · effort M

Clicking a help badge starts a tour directly while the guide/manual chooser lives in a hover tooltip. Disabling help removes BadgeHelp entirely and therefore removes most tour restart paths.

Plan: open a touch-friendly Help menu with Quick guide and Read manual, and keep a stable Guides entry independent of inline-help visibility.

P1

Narrow screens need a dedicated layout · effort M

Desktop 400 px popover positioning is clamped into small viewports and does not account for mobile visual viewport or safe-area changes.

Plan: use a compact bottom sheet and keep the target visible above it.

P1

UX outcomes are not measurable · effort M

The engine persists local status but emits no events for invitations, starts, exits, missing anchors, branches, completion, or time per step.

Plan: emit privacy-safe lifecycle events keyed by tour/version/step/route/locale and viewport class.

P1

Tests do not represent current tours or UX risks · effort M

E2E coverage is Russian desktop and Sandbox-centric. It expects three constituent-list steps although the tour now has five, and its seeded Sandbox version is six while the current tour is seven.

Plan: generate one smoke path per tour/anchor and add accessibility, mobile, delayed-anchor, load-error, route-resume, and read-only scenarios.

P2

Some guidance is stale or overly dense · effort S

Sandbox copy points to a menu without tab tours, graph steps read like shortcut manuals, and Russian content has grammar errors plus an untranslated “Details”.

Plan: correct all locales, copyedit Russian, and keep each step focused on one outcome and one action.

P2

Tooltip state is restored unconditionally · effort S

Ending a tour enables tooltips even if another flow had disabled them before the tour.

Plan: snapshot and restore the previous tooltip-enabled value.

Delivery plan

Wave 1 — Runtime correctness and truthful content

Small–mediumShip first
  • Clear stale anchor state on every tour and step transition.
  • Recover from lazy-load failures without leaving the application inert.
  • Raise anchor readiness tolerance and distinguish unavailable from still-loading.
  • Add explicit Close for now and clarify permanent Skip tour semantics.
  • Fix Sandbox re-entry copy, Russian copy issues, E2E step counts, and version fixtures.

Wave 2 — Accessibility and responsive behavior

MediumNo product analytics dependency
  • Associate dialog title/body and announce step changes.
  • Define keyboard behavior for modal and interactive steps.
  • Preserve focus and tooltip state across every exit path.
  • Honor reduced motion, zoom, mobile visual viewport, and safe-area insets.
  • Render a bottom sheet on narrow screens.

Wave 3 — Learning by doing

LargeRequires interaction model decision
  • Add mode: "explain" | "interact" to tour steps.
  • Add optional action completion predicates/events and target interaction boundaries.
  • Advance interactive steps only after the intended action succeeds.
  • Name subtours, show their length and breadcrumb, and support Return to overview.
  • Shorten the Sandbox overview and move shortcut inventories into manuals.

Wave 4 — Invitation, discovery, and measurement

MediumProduct validation
  • Replace immediate auto-start with an invitation shown after page readiness.
  • Keep Guides discoverable when inline help badges are disabled.
  • Provide a touch-friendly Guide / Manual chooser.
  • Instrument invitation, start, exit, unavailable target, branch, and completion events.
  • Use measured completion and exit points to revise step length and copy.

Acceptance criteria

Runtime and recovery

  • No tour load or anchor failure can leave the application inert without visible recovery controls.
  • A spotlight never displays the previous step's target.
  • Unavailable targets are disclosed and do not silently mark the tour complete.
  • Close for now preserves progress; Skip tour clearly records a deliberate opt-out.

Learning and accessibility

  • Interactive steps permit and verify the action described by their copy.
  • Screen readers announce the current title and “Step N of M”.
  • Keyboard focus stays predictable and returns safely after exit.
  • Reduced motion, 200% zoom, touch, and narrow screens remain usable.

Discovery and content

  • Every registered tour has a discoverable restart path even when inline help is hidden.
  • Sandbox instructions point to the actual contextual Guide entry.
  • Subtour actions name the destination and number of steps.
  • English, Russian, and French remain structurally aligned and copy-reviewed.

Verification and measurement

  • Every registered tour has an automated seeded smoke path.
  • Tests cover delayed/missing anchors, failed imports, read-only states, and route resume.
  • Lifecycle events distinguish invitation decline, temporary close, permanent skip, and completion.
  • Product owners can identify the most common exit and unavailable-target points.

Decisions required before Wave 3

Interaction boundary

Decide whether interactive steps unlock only the exact target, a containing region, or the full page. Exact-target interaction is safest; region interaction handles forms and graph canvases better.

Completion signals

Choose between DOM events, feature callbacks, or declarative predicates. Feature callbacks are explicit and robust but create more integration work.

Auto-start policy

Decide whether skipped users remain opted out across all future versions, and which changes qualify for a non-blocking What's new invitation.

Analytics destination

Select the existing product telemetry destination and define a privacy-safe event schema before adding instrumentation.

Strengths to preserve

State continuity

Per-tour status, versioning, resume points, route pause, and nested return behavior are well modeled.

Content architecture

Tour definitions remain data-oriented, lazy-loaded, colocated, and validated across three locales.

Contextual documentation

Help badges and manual links already connect lightweight guidance with deeper reference content.

Defensive async lifecycle

Loads and anchor waits are cancellable, routes are checked, and duplicate lazy loads are cached.
